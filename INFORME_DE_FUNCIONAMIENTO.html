INFORME DE FUNCIONAMIENTO Y ARQUITECTURA
Sistema de Mensajería Distribuida Java

Objetivo de este documento: Detallar el funcionamiento interno del sistema, el flujo de datos, manejo de concurrencia, persistencia, seguridad, y proveer la base técnica necesaria para el modelado UML y arquitectónico del proyecto.

1. Funcionamiento Interno y Flujo de Datos

1.1. Manejo de Concurrencia e Hilos (Threads)

El sistema utiliza un modelo avanzado de concurrencia basado en "Dos hilos por cliente" en el servidor, lo cual asegura que las operaciones de red de un usuario nunca afecten a los demás.

  • Hilo Principal (Servidor): El bucle principal del Servidor.java solo hace una cosa: hacer serverSocket.accept(). Cuando un cliente se conecta, delega el socket a un ManejadorCliente y crea un hilo nuevo para él.
  • Hilo Enviador (ManejadorCliente - run()): Lee constantemente el ObjectInputStream recibiendo mensajes del cliente. Su trabajo es "enrutar". Cuando recibe un mensaje, busca el buzón del destinatario en RAM y deposita el mensaje ahí de forma instantánea.
  • Hilo Receptor (ManejadorCliente - procesarBuzon()): Cada cliente tiene un hilo extra en el servidor dedicado EXCLUSIVAMENTE a extraer mensajes de su propio buzón e inyectarlos por la red hacia el cliente a través del ObjectOutputStream.

1.2. Uso de Búferes (Colas Sincronizadas) y el Slow Consumer Problem

El núcleo del enrutamiento es la estructura LinkedBlockingQueue<Mensaje> buzónMensajes. Esto soluciona el problema de los clientes con mala conexión (Slow Consumer Problem):

Si el usuario A tiene mala conexión, su Hilo Receptor en el servidor se quedará bloqueado intentando empujar los bytes por el socket. Sin embargo, si el usuario B le envía 100 mensajes, el Hilo Enviador de B simplemente dejará esos 100 mensajes en la LinkedBlockingQueue de A. Como la cola está en RAM, esta operación toma milisegundos y el hilo de B queda libre para seguir trabajando. El servidor nunca se congela por culpa de un cliente lento.

1.3. Solución a Condiciones de Carrera (Race Conditions)

Las condiciones de carrera se evitan separando estrictamente lo que es memoria RAM de lo que es Base de Datos:

  1. En RAM: El diccionario HashMap<String, ManejadorCliente> clientesConectados es el recurso crítico. Se protege usando un bloque synchronized(clientesConectados). Las operaciones dentro de este bloque son minúsculas (solo put, get, o remove), por lo que el candado se libera casi instantáneamente.
  2. En Base de Datos: Las consultas a MySQL NUNCA se hacen dentro del bloque synchronized. Si MySQL tarda en guardar un mensaje, esto ocurre fuera del candado, permitiendo que otros hilos sigan ruteando mensajes concurrentemente en la RAM. Además, GestorBD abre una nueva Connection por cada operación, delegando a MySQL el manejo transaccional.

1.4. Persistencia en Base de Datos (MySQL)

El sistema abandonó el almacenamiento en memoria volátil (Listas) y ahora utiliza el motor InnoDB de MySQL a través de la clase GestorBD (Patrón DAO):

  • usuarios: Guarda el nombre (UNIQUE) y el hash SHA-256 de la contraseña.
  • historial_mensajes: Guarda de forma permanente todos los mensajes ruteados en el sistema (contenido cifrado con AES).
  • mensajes_offline: Funciona como un buzón temporal (límite de 100 mensajes). Si el destinatario no está en el HashMap de conectados, el mensaje se inserta aquí. Al conectarse, el servidor los extrae y hace un DELETE.
  • Archivos Offline: Los archivos binarios evitan el límite max_allowed_packet de MySQL guardándose directamente en el disco duro del servidor (/server_files). En MySQL solo se guarda la ruta física cifrada. Al entregarse, el archivo se borra del disco.

2. Información para Diagramas (1.2 Modelado de Ingeniería)

Modelo Físico

  • Nodos: Computadoras de Escritorio / Laptops.
  • Entorno de Red: LAN/WAN. Los clientes requieren conexión TCP/IP hacia la IP del servidor.
  • Representación Gráfica: Diagrama de despliegue donde múltiples Nodos Cliente (con JRE) se conectan a través de una red (nube) a un Nodo Servidor Central. El Nodo Servidor está conectado localmente a un Nodo Base de Datos (MySQL/XAMPP).

Modelo Arquitectónico

  • Tipo de Arquitectura: Cliente-Servidor Centralizado (Topología en Estrella), Multihilo.
  • Capas:
    • Presentación (Frontend): Interfaces Swing (Cliente.java). Muestra datos y captura eventos.
    • Lógica de Negocio (Backend): Servidor.java y ManejadorCliente.java. Controla el ruteo, autenticación y listas de usuarios.
    • Persistencia de Datos: GestorBD.java (Capa DAO) comunicándose con MySQL vía JDBC.

Modelado de Funciones (Diagramas de Secuencia UML)

Función 1: Chat de Texto (Destinatario Online)

  1. Cliente Emisor envía Mensaje(TEXTO) por Socket.
  2. ManejadorCliente (Enviador) recibe y llama a Servidor.reenviarMensaje().
  3. Servidor busca en clientesConectados. Encuentra al receptor.
  4. Servidor inserta el mensaje en la LinkedBlockingQueue del receptor y llama a GestorBD.guardarMensajeHistorial().
  5. ManejadorCliente (Receptor) del destinatario saca el mensaje de la cola y lo envía por Socket.
  6. Cliente Receptor lo muestra en la UI.

Función 2: Transferencia de Archivo (Destinatario Offline)

  1. Cliente Emisor adjunta archivo y envía Mensaje(ARCHIVO) por Socket.
  2. Servidor no encuentra al destinatario en clientesConectados.
  3. Servidor verifica en GestorBD si el usuario existe. Sí existe.
  4. Servidor llama a GestorBD.guardarMensajeOffline().
  5. GestorBD extrae los bytes, los guarda en /server_files/ en el disco local, y guarda la ruta en MySQL (tabla mensajes_offline).
  6. Servidor notifica al Cliente Emisor que se guardó en buzón.

3. Información para Análisis (1.3 Análisis Fundamental)

Modelo de Seguridad

  • Canal Inseguro: La conexión TCP pura a través de Internet (WAN) es susceptible a ataques Man-in-the-Middle (MitM). Actualmente, los objetos viajan serializados pero no encriptados en tránsito (propuesta de mitigación: usar SSL/TLS con SSLSocket).
  • Seguridad en Reposo (Mitigación implementada): Para evitar fugas de información desde el lado del servidor, se ha implementado la clase Seguridad.java.
    • Contraseñas: No se guardan en texto plano. Se procesan por la función hash criptográfica SHA-256 (irreversible). Si la BD es comprometida, las credenciales no son reveladas.
    • Mensajes: El contenido de los mensajes de chat y las rutas de los archivos se cifran usando AES-128 (Cifrado Simétrico) antes de entrar a las tablas historial_mensajes y mensajes_offline. El administrador de la base de datos no puede leer las conversaciones.

Modelo de Fallos

  • Fallo por Crash (Caída de cliente): Si un proceso Cliente se cierra inesperadamente o pierde internet (EOFException / SocketException), el Hilo Enviador del servidor lo detecta, rompe el bucle, cierra los streams, y ejecuta el bloque finally donde llama a Servidor.removerCliente(), lo cual actualiza el estado del sistema limpiamente.
  • Fallo por Crash (Caída del servidor): Si el servidor cae, los clientes lo detectan en su HiloReceptor y la interfaz muestra "Desconectado". Los datos no se pierden, ya que las estructuras volátiles (HashMap) solo mantienen el estado de presencia. Toda la información dura está respaldada en InnoDB (transaccional).
  • Fallo por Omisión (Pérdida de paquetes red): Delegado a la capa TCP del sistema operativo, que garantiza entrega en orden, suma de verificación y retransmisión automática.
  • Fallo de Base de Datos: Si MySQL se apaga en tiempo de ejecución, el sistema captura la SQLException en la capa DAO. El sistema está diseñado para enviar una notificación visual al usuario (Ej: "Error al guardar el mensaje offline") permitiendo que el servidor principal continúe operando sin colapsar (Fail-safe parcial).
